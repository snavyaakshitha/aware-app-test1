## CRITICAL FINDING 1: Sugar Thresholds Based on Medical Guidelines

### WHO Guidelines (2015) - Global Standard
- **Strong recommendation**: Free sugars < 10% of total daily energy intake
- **Conditional recommendation**: Free sugars < 5% of total daily energy intake

**Per-Day Amounts:**
```
WHO Target (5% of daily calories):
- Average adult (2000 cal): 25g added sugar/day (6 teaspoons)
- Average child (1200 cal): 15g added sugar/day (3.75 teaspoons)
- Average toddler (900 cal): 11g added sugar/day (2.75 teaspoons)

WHO Upper Limit (10% of daily calories):
- Average adult (2000 cal): 50g added sugar/day (12 teaspoons)
- Average child (1200 cal): 30g added sugar/day (7.5 teaspoons)
```

### FDA Daily Value (2024 Update)
- **Daily Value for Added Sugars**: 50g/day (based on 2000 calorie diet)
- Products must show "% Daily Value" of added sugars
- FDA considers <5% DV "low", 20%+ DV "high"

**Practical FDA Numbers:**
```
Low added sugar: <5g/serving (10% DV)
Moderate: 5-10g/serving (10-20% DV)
High: 10-25g/serving (20-50% DV)
Very high: >25g/serving (>50% DV)
```

### American Heart Association (AHA) - Gender & Age Specific
**Adults:**
- Women: Max 25g/day (6 teaspoons, 100 calories)
- Men: Max 36g/day (9 teaspoons, 150 calories)

**Children:**
- Ages 2-18: Max 25g/day (6 teaspoons) — STRICT
- Ages 0-2: ZERO added sugars (breast milk/formula only)

**Cardiovascular Risk:**
- >10% daily calories from added sugar: 26% increased risk of CVD
- >17.5% daily calories: 38% increased CVD risk

## CRITICAL FINDING 3: Barcode Collision Detection

### What is a Barcode Collision?
Same barcode maps to 2+ different products in database (e.g., Purell hand sanitizer → Strawberry Rice Rusks)

### Why It Happens
- Open Food Facts: Community-sourced, unverified
- Barcode re-use: Products discontinued, barcodes recycled
- Data entry errors: Wrong barcode scanned/entered
- Format variations: EAN-13 vs UPC-A (different lengths, same product)
- Multi-pack barcodes: Different from individual product barcodes

### Detection Algorithm (Implementation)

**Step 1: Category Validation**
```
IF barcode_prefix IN [200-209] THEN category = 'cosmetics'
ELSE IF barcode_prefix IN [100-199] THEN category = 'restricted'
ELSE category = 'food' (default)

IF returned_product.category ≠ expected_category THEN
  FLAG_COLLISION = TRUE
  LOG: "Barcode [code] categorized as [category] but prefix suggests [expected]"
END
```

**Step 2: Product Name Matching**
```
IF user_scanned_image ≠ NULL THEN
  ocr_brand = OCR(image)  // Extract brand from photo
  returned_brand = product.brand
  
  IF similarity_score(ocr_brand, returned_brand) < 0.7 THEN
    FLAG_COLLISION = TRUE
    SHOW_USER: "Is this [scanned brand]? Our database shows [returned brand]. Confirm?"
  END
END
```

**Step 3: Data Quality Validation**
```
IF product.ingredients.length == 0 THEN
  FLAG: "Incomplete data — verify on package"
END

IF product.ingredients.length > 50 THEN
  FLAG: "Suspicious ingredient count — may be data error"
END

IF product.brand.length == 0 OR product.name.length == 0 THEN
  FLAG_COLLISION = TRUE
  LOG: "Missing brand/name for barcode [code]"
END
```

**Step 4: Timestamp & Source Check**
```
IF product.last_modified > 30_days_ago THEN
  // Recently updated, likely community fix
  confidence = HIGH
ELSE IF product.last_modified < 1_year_ago THEN
  confidence = MEDIUM
ELSE
  confidence = LOW
  FLAG: "Data not updated in 1+ year"
END
```

### User Experience: Collision Detection Flow

**Scenario: User scans Purell hand sanitizer**

```

### Data Enrichment: Multi-Source Validation

**Smart Matching:**
```
1. Query USDA FDC (US products)
2. If not found → Query OFF
3. If found in both → Compare:
   - Brand match?
   - Ingredient overlap?
   - Nutrient data alignment?
   - Timestamp freshness?
4. If OFF data suspicious → Flag for manual review

```

## CRITICAL FINDING 4: Comprehensive Hallucination Prevention

### Root Cause #1: AI Image Quality (DETAILED ANALYSIS)

**When Gemini/GPT Hallucinate:**
- Image unreadable (blurry, dark, wrong angle)
- Label partially visible (cropped)
- Background noise (competing text)
- Non-English labels (no English fallback)

**Prevention Strategy:**

#### A. Image Quality Validation
```
IMAGE_QUALITY_CHECKS:
1. Sharpness score (Laplacian variance)
   - Good: >100
   - Fair: 50-100
   - Poor: <50
   
2. Brightness/contrast
   - Reject if <30 or >200 on 0-255 scale
   
3. Text detectability
   - Run OCR, must extract >50% of label text
   - If <50% → Reject with "Image too unclear"
   
4. Label completeness
   - Must show >60% of label (top to bottom)
   - Warn if <80% visible
   
5. Barcode visibility
   - Barcode must be readable or visible
   - If neither → Warn user to include barcode
```

**User Flow:**
```
[Take photo of nutrition label]

IMAGE QUALITY CHECK:
✓ Sharpness: 145 (Good)
✓ Brightness: Optimal
✓ Text visible: 78%
⚠️ Barcode: Partially cut off (not critical)

[PROCEED to AI extraction]
```

#### B. Gemini Confidence Scoring
```
Gemini response includes confidence for each field:

{
  "product_name": {
    "value": "OMG Gluten Free Cookie",
    "confidence": 0.92  // High
  },
  "ingredients": {
    "value": ["..."],
    "confidence": 0.45  // LOW — FLAG FOR REVIEW
  },
  "nutrition": {
    "value": {...},
    "confidence": 0.88  // High
  }
}

RULE:
IF confidence < 0.70 THEN
  FLAG for manual review
  SHOW user: "Could not extract reliably. Please verify or use manual entry."
END
```

#### C. Ingredient Count Sanity Check
```
INGREDIENT_COUNT_VALIDATION:
- Typical products: 3-20 ingredients
- High-complexity products: 20-40 ingredients
- Ultra-processed: 40-60 ingredients
- Hallucination threshold: >80 ingredients

IF ingredient_count > 80 THEN
  ALERT: "Suspicious ingredient count detected"
  FLAG for manual review
  SHOW user: "Could not extract ingredient list reliably"
END

IF ingredient_count > 60 AND confidence < 0.8 THEN
  FLAG for manual review
END
```

#### D. Ingredient Validation Against Known Database
```
for each extracted_ingredient in ingredients:
  IF ingredient NOT IN food_database THEN
    confidence_adjustment -= 0.1
  END
  
  IF ingredient matches suspicious_patterns THEN
    // e.g., "naphthylamines", "hand sanitizer", "industrial chemical"
    FLAG_HALLUCINATION = TRUE
    LOG: "Suspicious ingredient detected: [ingredient]"
  END
END

IF FLAG_HALLUCINATION THEN
  REJECT extraction
  SHOW: "This doesn't look like a food product. Is the photo correct?"
END
```

### Root Cause #2: Barcode Collision (DETAILED SOLUTION)

**Implementation (added to Phase 2/3):**

1. **Barcode Prefix Validation** (happens before API call)
```
barcode = "0284000034951"  // Purell example

prefix = barcode[0:2]
IF prefix IN ["00-06", "30-37"] THEN expected_category = "general"
IF prefix IN ["20-29"] THEN expected_category = "restricted/cosmetics"

returned_product = query_OFF(barcode)

IF returned_product.category ≠ expected_category THEN
  collision_risk = HIGH
  request_user_confirmation = TRUE
END
```

2. **Product Name/Brand Matching**
```
IF user_image ≠ NULL THEN
  visible_brand = OCR_extract_brand(image)
  returned_brand = returned_product.brand
  
  // Fuzzy matching (allows for slight differences)
  similarity = fuzzy_match(visible_brand, returned_brand)
  
  IF similarity < 0.6 THEN
    // Likely collision
    SHOW_CONFIRMATION_DIALOG:
    "You scanned [visible]: Is this correct?"
    [returned_product.image] → [YES / NO]
  END
END
```

3. **Data Completeness Check**
```
IF returned_product.ingredients.count == 0 THEN
  incomplete_risk = HIGH
  SHOW: "Ingredient data incomplete — verify on package"
END

IF returned_product.nutriments == NULL THEN
  incomplete_risk = HIGH
  SHOW: "Nutrition data missing — checking other sources"
END
```

4. **Temporal Validation**
```
last_updated = returned_product.updated_at
days_old = NOW() - last_updated

IF days_old > 365 THEN
  data_freshness = LOW
  LOG: "Product data not updated in 1+ year"
  SHOW: "Data may be outdated. Verify key info on package."
END
```

### Root Cause #3: OFF Data Quality (DETAILED SOLUTION)

**Problems in Open Food Facts:**
- ~40-60% ingredient completeness for many products
- Barcode collisions (12,000+ problematic codes)
- Community-sourced, unverified
- No data quality validation

**Mitigation:**

1. **Cross-Source Validation**
```
query_result = query_OFF(barcode)

IF query_result.completeness < 70% THEN
  // Try alternative sources
  usda_result = query_USDA(product_name)
  
  IF usda_result ≠ NULL THEN
    // Merge data, use USDA for nutrition
    merged = {
      ...query_result,
      nutrients: usda_result.nutrients,
      confidence: "USDA validated"
    }
  ELSE
    // Flag as incomplete
    SHOW: "Incomplete ingredient data. Verify on package."
  END
END
```

2. **User Correction Loop**
```
SHOW_FEEDBACK_BUTTONS:
[✓ Accurate] [⚠️ Something wrong] [? Missing info]

IF user_clicks "Something wrong" THEN
  SHOW_FORM: {
    product_name: [editable],
    brand: [editable],
    ingredients: [editable],
    notes: [comment]
  }
  
  SUBMIT to manual_review_queue
  LOG for data enrichment
  NOTIFY: OFF/USDA maintainers
END
```

3. **Confidence Scoring for Each Field**
```
